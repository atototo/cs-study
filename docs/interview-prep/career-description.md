# 💼 경력기술 및 프로젝트 경험 준비

경력기술서와 면접에서 프로젝트 경험을 효과적으로 전달하기 위한 준비 가이드입니다. 
단순히 "무엇을 했다"가 아니라, **"어떤 문제가 있었고, 왜 그런 선택을 했으며, 그 결과가 어떠했는지"**를 중심으로 정리하는 것이 핵심입니다.

## 1. 프로젝트 경험 정리 프레임워크 (STAR+)

### 🏗 프로젝트 개요 (Overview)
- **프로젝트명**:
- **기간**:
- **인원 및 역할**:
- **사용 기술(Tech Stack)**:

### 🚩 문제 및 배경 (Situation & Task)
* 프로젝트를 시작하게 된 배경은 무엇인가요?
* 해결하고자 했던 핵심 문제는 무엇인가요?
* 당시 비즈니스적/기술적 제약 사항은 무엇이었나요?

### 💡 해결 과정 및 고려사항 (Action & Consideration)
* **어떤 기술적 의사결정을 내렸나요?** (Why this tech?)
* **대안은 무엇이었나요?** (Alternative)
* **해당 방식을 선택한 논리적 근거는 무엇인가요?** (Trade-off 고려)
* 구현 과정에서 부딪힌 구체적인 어려움과 해결책은?

### 📈 결과 및 성과 (Result)
* **정량적 성과**: (예: 로딩 속도 50% 개선, 트래픽 처리량 2배 증가 등)
* **정성적 성과**: (예: 팀 내 코드 리뷰 문화 정착, 유지보수성 향상 등)
* **배운 점 (Lesson Learned)**: 아쉬웠던 점이나 개선하고 싶은 점

## 2. 주요 질문 리스트 (Self-Check)

작성한 내용을 바탕으로 다음 질문들에 답변할 수 있는지 점검해보세요.

### 🛠 기술적 깊이 관련
- 이 기술을 선택한 가장 큰 이유는 무엇인가요? 다른 대안과 비교했을 때의 장점은?
- 프로젝트 중 가장 기술적으로 어려웠던 트러블슈팅 경험 하나만 이야기해주세요.
- 해당 아키텍처의 한계점은 무엇이며, 더 개선한다면 어떻게 할 수 있을까요?

### 🤝 협업 및 프로세스 관련
- 기획이나 디자인 팀과 의견 충돌이 있었던 적이 있나요? 어떻게 해결했나요?
- 일정 압박이 심했을 때 어떻게 우선순위를 조정을 했나요?
- 팀 내에서 본인이 주도적으로 개선한 프로세스나 문화가 있나요?

## 3. 작성 예시 (Template)

### [예시] 실시간 채팅 서비스 성능 개선

#### Situation (배경)
- 기존 폴링(Polling) 방식의 채팅 서비스로 서버 부하 증가 및 메시지 지연 발생
- 동시 접속자 1,000명 이상 시 응답 속도가 3초 이상 지연됨

#### Task (과제)
- 실시간성을 보장하고 서버 리소스를 효율적으로 사용하는 구조로 변경 필요

#### Action (해결 & 고려사항)
- **기술 선정**: WebSocket vs SSE 고려 중, 양방향 통신이 필수적이라 Socket.io 도입
- **문제 해결**: 
    - 메시지 유실 방지를 위해 Redis Pub/Sub 도입하여 다중 서버 환경 대응
    - 불필요한 재연결 요청을 막기 위해 Exponential Backoff 알고리즘 적용

#### Result (결과)
- 메시지 전송 지연 시간 3초 -> 0.1초 미만으로 단축
- 서버 CPU 점유율 40% 감소
- 동시 접속자 3,000명까지 안정적으로 처리 가능 확인

## 4. 팁 (Tips)
- 수치(%)로 표현할 수 있는 부분은 최대한 구체적으로 적으세요.
- **Fail Story**도 좋습니다. 실패에서 무엇을 배웠고 어떻게 극복했는지는 훌륭한 스토리텔링이 됩니다.
- "우리"가 아닌 "**나**"의 기여도를 명확히 하세요.

### 📂 [프로젝트 1: 근태/업무관리 시스템 멀티테넌시 구조 전환]

- **기간**: 2024.10 ~ 2025.06
- **역할**: Backend Developer(기여도 100% / 설계 주도 및 리드)
- **Tech Stack**: Java 21, Spring Boot 3.1, Spring Batch, JPA, QueryDSL, Redis, Kubernetes, GitAction, ArgoCD

#### 배경 
SaaS 전환을 위한 사업 방향에 따라 멀티테넌시 지원이 필요했으나, 기존 시스템이 사번(Natural Key)을 PK로 사용하는 구조여서 법인 간 데이터 충돌 위험이 있었음. 또한 인사/계정 API에서 받은 데이터를 도메인 테이블에 직접 반영하는 구조로, 동기화 실패 시 데이터 손상 및 복구가 어려운 상황이었음.  

#### 핵심 과제
1. PK 재설계 및 멀티테넌시 구조 전환
2. HR 연동 파이프라인 재설계
3. 통계 조회 성능 최적화

#### 상세 해결 과정
- [1] PK 재설계 및 멀티테넌시 구조 전환
    - 사번 기반 Natural Key를 Surrogate Key로 전환
        - 법인 확장 시 사번 중복으로 인한 데이터 충돌 방지
        - API 파라미터에 사번 직접 노출로 인한 보안 우려 해소
    - 모든 엔티티에 tenantId, orgCode 필수 컬럼 추가하여 법인별 데이터 격리
    - 기존 기능 보장을 위해 ASIS 데이터 기반 검증: 개발자 E2E 테스트 및 운영 데이터 1개월치 마이그레이션 후 통계 정합성 확인

- [2] HR 연동 파이프라인 재설계
    - 기존 문제: 인사마스터 → 도메인 테이블 직접 동기화로 실패 시 데이터 손상 및 원천 데이터 미보존
    - 해결: 2-Step 동기화 구조 도입
        - Step 1: 인사마스터 → Staging Table(SYNC_*) 원천 데이터 저장
        - Step 2: 검증 후 도메인 Table(JOB_*) 반영
        - 원천 동기화 성공한 법인만 도메인 동기화 진행하여 단계별 실패 격리
    - 추가 처리 
        - 데이터 표현 규약 차이 대응 (퇴직일자: null → 2999-12-31)
        - DELETE 후 INSERT → Upsert 방식 전환 (PK 참조 테이블 정합성 보장, DDL 권한 제약으로 임시 테이블 스왑 방식 제외)    

- [3] 통계 조회 성능 최적화
    - 문제: 작업 검증 중 통계 페이지 Timeout 발생. SELECT 절 내 다중 서브쿼리로 100명 조회 시 2,400회 이상 쿼리 발생
    - 해결: CaseBuilder + GROUP BY 조건부 집계로 쿼리 단일화, fetchJoin()으로 연관 엔터티 로딩 최적화, 복합 인덱스 설계 적용    
    - 캐싱/비동기 처리는 대규모 구조 변경 직후 안정성 확보 우선으로 순차 도입 계획    

#### 결과
- 비즈니스 성과: 단일 시스템에서 4개 법인 데이터 통합 관리 및 안정적 운영
- 확장성 개선: 신규 법인 추가 시 코드 수정 불필요
- 성능 개선: 통계 페이지 쿼리 수 2,400회 → 4회, 조회 속도 Timeout → 0.5초 이내
