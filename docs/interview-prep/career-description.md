# 💼 경력기술 및 프로젝트 경험 준비

경력기술서와 면접에서 프로젝트 경험을 효과적으로 전달하기 위한 준비 가이드입니다. 
단순히 "무엇을 했다"가 아니라, **"어떤 문제가 있었고, 왜 그런 선택을 했으며, 그 결과가 어떠했는지"**를 중심으로 정리하는 것이 핵심입니다.

## 1. 프로젝트 경험 정리 프레임워크 (STAR+)

### 🏗 프로젝트 개요 (Overview)
- **프로젝트명**:
- **기간**:
- **인원 및 역할**:
- **사용 기술(Tech Stack)**:

### 🚩 문제 및 배경 (Situation & Task)
* 프로젝트를 시작하게 된 배경은 무엇인가요?
* 해결하고자 했던 핵심 문제는 무엇인가요?
* 당시 비즈니스적/기술적 제약 사항은 무엇이었나요?

### 💡 해결 과정 및 고려사항 (Action & Consideration)
* **어떤 기술적 의사결정을 내렸나요?** (Why this tech?)
* **대안은 무엇이었나요?** (Alternative)
* **해당 방식을 선택한 논리적 근거는 무엇인가요?** (Trade-off 고려)
* 구현 과정에서 부딪힌 구체적인 어려움과 해결책은?

### 📈 결과 및 성과 (Result)
* **정량적 성과**: (예: 로딩 속도 50% 개선, 트래픽 처리량 2배 증가 등)
* **정성적 성과**: (예: 팀 내 코드 리뷰 문화 정착, 유지보수성 향상 등)
* **배운 점 (Lesson Learned)**: 아쉬웠던 점이나 개선하고 싶은 점

## 2. 주요 질문 리스트 (Self-Check)

작성한 내용을 바탕으로 다음 질문들에 답변할 수 있는지 점검해보세요.

### 🛠 기술적 깊이 관련
- 이 기술을 선택한 가장 큰 이유는 무엇인가요? 다른 대안과 비교했을 때의 장점은?
- 프로젝트 중 가장 기술적으로 어려웠던 트러블슈팅 경험 하나만 이야기해주세요.
- 해당 아키텍처의 한계점은 무엇이며, 더 개선한다면 어떻게 할 수 있을까요?

### 🤝 협업 및 프로세스 관련
- 기획이나 디자인 팀과 의견 충돌이 있었던 적이 있나요? 어떻게 해결했나요?
- 일정 압박이 심했을 때 어떻게 우선순위를 조정을 했나요?
- 팀 내에서 본인이 주도적으로 개선한 프로세스나 문화가 있나요?

## 3. 작성 예시 (Template)

### [예시] 실시간 채팅 서비스 성능 개선

#### Situation (배경)
- 기존 폴링(Polling) 방식의 채팅 서비스로 서버 부하 증가 및 메시지 지연 발생
- 동시 접속자 1,000명 이상 시 응답 속도가 3초 이상 지연됨

#### Task (과제)
- 실시간성을 보장하고 서버 리소스를 효율적으로 사용하는 구조로 변경 필요

#### Action (해결 & 고려사항)
- **기술 선정**: WebSocket vs SSE 고려 중, 양방향 통신이 필수적이라 Socket.io 도입
- **문제 해결**: 
    - 메시지 유실 방지를 위해 Redis Pub/Sub 도입하여 다중 서버 환경 대응
    - 불필요한 재연결 요청을 막기 위해 Exponential Backoff 알고리즘 적용

#### Result (결과)
- 메시지 전송 지연 시간 3초 -> 0.1초 미만으로 단축
- 서버 CPU 점유율 40% 감소
- 동시 접속자 3,000명까지 안정적으로 처리 가능 확인

## 4. 팁 (Tips)
- 수치(%)로 표현할 수 있는 부분은 최대한 구체적으로 적으세요.
- **Fail Story**도 좋습니다. 실패에서 무엇을 배웠고 어떻게 극복했는지는 훌륭한 스토리텔링이 됩니다.
- "우리"가 아닌 "**나**"의 기여도를 명확히 하세요.

## 5. 내 프로젝트 경험 (작성란)

### 📂 [프로젝트 1: 근태/업무관리 시스템 멀티테넌시 구조 전환]

#### Overview
- **기간**: 2024.10 ~ 2025.06
- **역할**: Backend Developer, 설계 주도 및 리드
- **Tech Stack**: Java 21, Spring Boot 3.1, Spring Batch, JPA, QueryDSL, Redis, Kubernetes, GitAction, ArgoCD

#### Situation (배경)
- 전사 비즈니스 확장을 위해 다중 법인(Multi-Tenancy) 수용이 필수적이었으나, 기존 시스템은 단일 법인에 강하게 결합된(Tightly Coupled) 아키텍처로 확장이 불가능한 상태

#### Task (과제)
- 핵심 1 (DB 구조): 사번 등 비즈니스 속성에 의존하는 내추럴 키(Natural Key) 기반의 PK 구조를 탈피하여, 데이터 중복 및 무결성 이슈 없이 N개 법인을 수용할 수 있는 DB 스키마 재설계
- 핵심 2 (마이그레이션): 운영 중인 대규모 데이터의 정합성을 훼손하지 않고, 신규 아키텍처로 안전하게 이관(Migration)하는 전략 수립
- 핵심 3 (연동): 이기종 인사 시스템(HR)과의 데이터 동기화 파이프라인 구축

#### Action (해결 & 고려사항)
- 비식별 관계(Non-identifying) 설계로 결합도 제거
    - 기존 내추럴 키(사번+부서)의 보안 취약점과 확장성 한계를 분석.
    - 복합키(Composite Key) 방식과 비교했을 때, **JPA 연관 관계의 복잡도를 낮추고 비즈니스 로직과 식별자를 분리(Decoupling)**하기 위해 UUID 기반의 Surrogate Key(대리키) 도입 결정.

- 대용량 데이터 처리를 위한 'Cut-off' 이관 전략 수립
    - 수백만 건의 통계 데이터 전체 이관 시 발생할 DB Lock 및 서비스 다운타임 리스크를 식별.
    - '월 마감'이라는 도메인 특성을 활용, 구 데이터는 조회 전용(Archive)으로 분리하고 신규 데이터부터 새 아키텍처를 적용하는 Cut-off 전략을 통해 운영 리스크를 90% 이상 감소시킴.

- 쿼리 성능 최적화 및 인덱스 재설계
    - PK 변경으로 인한 실행 계획(Execution Plan) 변화를 분석하여, 통계 집계에 최적화된 커버링 인덱스(Covering Index) 적용.
    - 복잡한 연관 관계로 발생한 N+1 문제를 식별하고, QueryDSL의 fetchJoin() 및 데이터 집계 로직 튜닝으로 조회 속도 개선.


#### Result (결과)
- 비즈니스 성과: "단일 시스템에서 4개 이상 법인 데이터 통합 관리 구현 및 안정적 운영."
- 기술적 성과: "PK 의존성 제거로 신규 법인 추가 시 소요 시간 00일 -> 0일로 단축(코드 수정 불필요)."
- 성능 개선: 통계 페이지 조회 속도 **Timeout(무응답) → 1초 이내(95% 단축)**로 개선하며 안정적인 서비스 오픈
