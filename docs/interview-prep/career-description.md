# 💼 경력기술 및 프로젝트 경험 준비

경력기술서와 면접에서 프로젝트 경험을 효과적으로 전달하기 위한 준비 가이드입니다. 
단순히 "무엇을 했다"가 아니라, **"어떤 문제가 있었고, 왜 그런 선택을 했으며, 그 결과가 어떠했는지"**를 중심으로 정리하는 것이 핵심입니다.

## 1. 프로젝트 경험 정리 프레임워크 (STAR+)

### 🚩 문제 및 배경 (Situation & Task)
* 프로젝트를 시작하게 된 배경은 무엇인가요?
* 해결하고자 했던 핵심 문제는 무엇인가요?
* 당시 비즈니스적/기술적 제약 사항은 무엇이었나요?

### 💡 해결 과정 및 고려사항 (Action & Consideration)
* **어떤 기술적 의사결정을 내렸나요?** (Why this tech?)
* **대안은 무엇이었나요?** (Alternative)
* **해당 방식을 선택한 논리적 근거는 무엇인가요?** (Trade-off 고려)
* 구현 과정에서 부딪힌 구체적인 어려움과 해결책은?

### 📈 결과 및 성과 (Result)

* **정성적 성과**: (예: 팀 내 코드 리뷰 문화 정착, 유지보수성 향상 등)
* **배운 점 (Lesson Learned)**: 아쉬웠던 점이나 개선하고 싶은 점

## 2. 주요 질문 리스트 (Self-Check)

작성한 내용을 바탕으로 다음 질문들에 답변할 수 있는지 점검해보세요.

### 🛠 기술적 깊이 관련
- 이 기술을 선택한 가장 큰 이유는 무엇인가요? 다른 대안과 비교했을 때의 장점은?
- 프로젝트 중 가장 기술적으로 어려웠던 트러블슈팅 경험 하나만 이야기해주세요.
- 해당 아키텍처의 한계점은 무엇이며, 더 개선한다면 어떻게 할 수 있을까요?

### 🤝 협업 및 프로세스 관련
- 기획이나 디자인 팀과 의견 충돌이 있었던 적이 있나요? 어떻게 해결했나요?
- 일정 압박이 심했을 때 어떻게 우선순위를 조정을 했나요?
- 팀 내에서 본인이 주도적으로 개선한 프로세스나 문화가 있나요?

## 3. 작성 예시 (Template)

### [예시] 실시간 채팅 서비스 성능 개선

#### Situation (배경)
- 기존 폴링(Polling) 방식의 채팅 서비스로 서버 부하 증가 및 메시지 지연 발생
- 동시 접속자 1,000명 이상 시 응답 속도가 3초 이상 지연됨

#### Task (과제)
- 실시간성을 보장하고 서버 리소스를 효율적으로 사용하는 구조로 변경 필요

#### Action (해결 & 고려사항)
- **기술 선정**: WebSocket vs SSE 고려 중, 양방향 통신이 필수적이라 Socket.io 도입
- **문제 해결**: 
    - 메시지 유실 방지를 위해 Redis Pub/Sub 도입하여 다중 서버 환경 대응
    - 불필요한 재연결 요청을 막기 위해 Exponential Backoff 알고리즘 적용

#### Result (결과)
- 메시지 전송 지연 시간 3초 -> 0.1초 미만으로 단축
- 서버 CPU 점유율 40% 감소
- 동시 접속자 3,000명까지 안정적으로 처리 가능 확인

<summary>근태/업무관리 시스템 멀티테넌시 구조 전환</summary>
### 📂 [프로젝트 1: 근태/업무관리 시스템 멀티테넌시 구조 전환]

- **기간**: 2024.10 ~ 2025.06
- **역할**: Backend Developer(기여도 100% / 설계 주도 및 리드)
- **Tech Stack**: Java 21, Spring Boot 3.1, Spring Batch, JPA, QueryDSL, Redis, Kubernetes, GitAction, ArgoCD

#### 배경 
SaaS 전환을 위한 사업 방향에 따라 멀티테넌시 지원이 필요했으나, 기존 시스템이 사번(Natural Key)을 PK로 사용하는 구조여서 법인 간 데이터 충돌 위험이 있었음. 또한 인사/계정 API에서 받은 데이터를 도메인 테이블에 직접 반영하는 구조로, 동기화 실패 시 데이터 손상 및 복구가 어려운 상황이었음.  

#### 핵심 과제
1. PK 재설계 및 멀티테넌시 구조 전환
2. HR 연동 파이프라인 재설계
3. 통계 조회 성능 최적화

#### 상세 해결 과정
- [1] PK 재설계 및 멀티테넌시 구조 전환
    - 사번 기반 Natural Key를 Surrogate Key로 전환
        - 법인 확장 시 사번 중복으로 인한 데이터 충돌 방지
        - API 파라미터에 사번 직접 노출로 인한 보안 우려 해소
    - 모든 엔티티에 tenantId, orgCode 필수 컬럼 추가하여 법인별 데이터 격리
    - 기존 기능 보장을 위해 ASIS 데이터 기반 검증: 개발자 E2E 테스트 및 운영 데이터 1개월치 마이그레이션 후 통계 정합성 확인

- [2] HR 연동 파이프라인 재설계
    - 기존 문제: 인사마스터 → 도메인 테이블 직접 동기화로 실패 시 데이터 손상 및 원천 데이터 미보존
    - 해결: 2-Step 동기화 구조 도입
        - Step 1: 인사마스터 → Staging Table(SYNC_*) 원천 데이터 저장
        - Step 2: 검증 후 도메인 Table(JOB_*) 반영
        - 원천 동기화 성공한 법인만 도메인 동기화 진행하여 단계별 실패 격리
    - 추가 처리 
        - 데이터 표현 규약 차이 대응 (퇴직일자: null → 2999-12-31)
        - DELETE 후 INSERT → Upsert 방식 전환 (PK 참조 테이블 정합성 보장, DDL 권한 제약으로 임시 테이블 스왑 방식 제외)    

- [3] 통계 조회 성능 최적화
    - 문제: 작업 검증 중 통계 페이지 Timeout 발생. SELECT 절 내 다중 서브쿼리로 100명 조회 시 2,400회 이상 쿼리 발생
    - 해결: CaseBuilder + GROUP BY 조건부 집계로 쿼리 단일화, fetchJoin()으로 연관 엔터티 로딩 최적화, 복합 인덱스 설계 적용    
    - 캐싱/비동기 처리는 대규모 구조 변경 직후 안정성 확보 우선으로 순차 도입 계획    

#### 결과
- 비즈니스 성과: 단일 시스템에서 4개 법인 데이터 통합 관리 및 안정적 운영
- 확장성 개선: 신규 법인 추가 시 코드 수정 불필요
- 성능 개선: 통계 페이지 쿼리 수 2,400회 → 4회, 조회 속도 Timeout → 0.5초 이내

### 📂 프로젝트 3: MCP 서버 개발 (PlayMCP 플랫폼)

#### Overview
- **기간**: 2025.09 ~ 현재
- **역할**: Backend Developer (기여도 100%)
- **Tech Stack**: TypeScript, Python, Node.js, MCP SDK, MySQL, Docker, Kubernetes, OpenTelemetry, Sentry

#### 배경
- 카카오 PlayMCP 플랫폼 대상 TO 계약으로 4개 MCP 서버 개발. 초기 병원/약국 정보 조회 MCP 개발 완료 후 공공데이터 서버 장애로 추가 개발 정체 → 수동 대기 대신 오픈 API 리서치 후 NASA 기반 신규 MCP 서버 기획 제안, 2개 프로젝트로 확장.

#### 개발 범위
- **OX Quiz**: 실시간 퀴즈 게임 서버 (세션 기반 게임 진행 관리)
- **병원/약국 정보 조회**: 공공데이터포털 연동 의료기관/약국 검색 (14개 도구)
- **재난 정보 조회**: GDACS/NASA 연동 글로벌 재난 정보 조회
- **우주 이미지/소행성 데이터 조회**: NASA API 연동 우주 이미지/소행성 데이터 조회

#### 핵심 과제
- 외부 API 이중화 및 Fallback 설계
- LLM 친화적 도구 인터페이스 설계
- Multi-Pod 환경의 Stateless 아키텍처 전환

#### 상세 해결 과정
- [1] 외부 API 이중화 및 Fallback 설계 (재난 정보 조회)
    - 기존 문제: NASA API 단독 의존 → MVP 검증 단계에서 데이터 부족 확인
    - 해결: 재난 특화 API(GDACS) 메인 + NASA Fallback 구조로 전환
        - 이중 소스 응답 정규화로 일관된 데이터 포맷 제공
        - 주 소스 실패 시 자동 Fallback 처리
    - 판단 근거: 단일 소스 의존 리스크 분산, 데이터 풍부도 확보
- [2] LLM 친화적 도구 인터페이스 설계 (전체 서버 공통)
    - 기존 문제: 플랫폼 제약(1회 8,000자, 10회 호출) + 서버가 사용자 의도 파악 불가하여 불필요한 API 확장 호출 발생
    - 해결: 제어권을 AI에게 위임하는 구조 설계
        - 응답 최적화 (~800 → ~400 bytes)로 페이징 효율화
        - 서버사이드 자동 재시도 → LLM 판단 기반 재시도로 전환
        - 시나리오별 도구 선택 가이드, 성공/실패 예시, 미지원 케이스 명시
    - 판단 근거: 서버는 "최근 7일" vs "지난 1월" 같은 사용자 의도 구분 불가 → AI가 맥락 파악 후 재조회 여부 판단하도록 설계
- [3] Multi-Pod 환경 세션 공유 문제 해결 (OX Quiz)
    - 기존 문제: 단일 Pod In-memory 세션 → Multi-Pod 배포 시 LB 라우팅으로 세션 유실 발생
    - 1차 시도 (실패): SQLite WAL + NFS 공유 볼륨 + Sticky Session → MCP HTTP 프로토콜 특성상 불안정
    - 최종 해결: MySQL 세션 영속화 + Stateless HTTP 전환 (매 요청 새 transport)
    - 판단 근거: 이미 MySQL 사용 중 + 세션 유지시간 1분 제한 → Redis 대비 인프라 복잡도 최소화

#### 결과
- 4개 MCP 서버 출시 (OX Quiz, Medical Radar, Disaster, Space)
- 외부 API 장애 대응력 확보 (Fallback 구조)
- Pod 간 상태 공유 없이 수평 확장 가능한 아키텍처 완성
- OpenTelemetry 자동 계측 + Sentry 연동으로 실시간 모니터링 체계 구축

### 📂 프로젝트 4: 금융권 실시간 알림 시스템 마이그레이션 (Flex → Node.js)

#### Overview
- **기간**: 2020.10 ~ 2021.01
- **역할**: Backend Developer (기여도 100%, 단독 설계/구현/배포)
- **Tech Stack**: Node.js 10.x, Socket.IO, Express, CentOS 6.9, Java(Spring), JSP

#### 배경
- 기존 여신 시스템의 실시간 알림/문서함 카운터가 Adobe Flex 기반으로 구현되어 있었음
- Flex 지원 종료 예정 + IE 전용 환경의 한계로 현대화 필요
- 폐쇄망(오프라인) 환경이라 npm/yum 사용 불가

#### 핵심 과제
- 폐쇄망 환경에서 Node.js 실행 환경 구축
- 인수인계 문서 없이 기존 Flex 통신 구조 파악
- 직원별 개인화된 실시간 알림/카운터 전송 시스템 설계

#### 상세 해결 과정
- [1] 폐쇄망 환경 구축
    - 기존 문제: npm/yum 사용 불가, rpm으로 설치하니 Node.js 버전이 너무 낮음
    - 해결: 소스 컴파일 방식으로 의존성 수동 해결
        - GCC 버전업 → Python 버전업 → Node.js 10.x 설치 (의존성 체인)
        - 포트 권한 이슈 해결 (1024 이하 포트 사용 불가 → 포트 변경)
        - 방화벽 정책 설정 및 nohup 백그라운드 실행 구성
    - 판단 근거: 로컬 PC에서 검색하며 하나씩 해결, 실서버 환경과 동일한 CentOS 6.9 가상머신에서 사전 검증
- [2] 레거시 역분석
    - 기존 문제: 인수인계 문서 부족, Flex 코드 구조 파악 필요
    - 해결: 화면(JSP)부터 역추적하여 전체 흐름 파악
        - JSP의 Socket.IO 클라이언트 연결 코드 → 이벤트 타입 확인
        - Java 백엔드(NodeManagerImpl)의 HTTP 통신 로직 분석
        - 알람/문서함 카운터 각각의 파라미터 구조 도출
    - 판단 근거: 팀장님께 질문하며 핵심 기능(알람, 문서함 건수) 우선순위 파악 후 집중 분석
- [3] Socket.IO 기반 실시간 시스템 구현
    - 기존 문제: 직원별로 본인 담당 건수만 받아야 함 → 접속자 식별/관리 필요
    - 해결: 사용자 ID 기반 소켓 등록/매칭 시스템 구축
        - 연결 시 사용자 ID로 소켓 등록 (`registerUser`)
        - 연결 해제 시 자동 정리 (`procDisconnect`)
        - 이벤트 수신 후 재등록 로직으로 연결 끊김 대비
        - 라우팅 분기 (`/sendToNode`, `/alarm`)로 이벤트 타입별 처리
    - 판단 근거: 연결 안정성 이슈 발생 → 재등록 로직 강화로 해결
